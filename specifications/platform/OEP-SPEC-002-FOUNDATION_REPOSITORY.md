# OEP-SPEC-002

# Foundation Repository Specification

**Version:** 1.0
**Status:** Draft

---

# 1. Purpose

This specification defines the canonical repository structure generated by **OEP Foundation**.

It is the authoritative blueprint used by the OEP CLI when creating a new repository.

Example:

```bash
oep init my-workshop
```

The generated repository shall conform to this specification unless an approved repository template explicitly extends it.

This specification defines repository structure only.

Implementation details are intentionally excluded.

---

# 2. Scope

This specification governs:

* Repository directory structure
* Standard files
* Configuration files
* Build system organization
* Workspace organization
* Placeholder generation
* Repository initialization behavior

This specification does not govern:

* Runtime implementation
* SDK implementation
* Studio implementation
* Registry implementation

---

# 3. Design Principles

Every generated repository shall be:

* Predictable
* Cross-platform
* Offline-first
* Version controlled
* Human-readable
* AI-friendly
* IDE-friendly
* Extensible

Consistency is preferred over customization.

---

# 4. Repository Types

Initial implementation supports a single repository type.

| Repository Type        | Status    |
| ---------------------- | --------- |
| Standard Repository    | Supported |
| Enterprise Repository  | Future    |
| Educational Repository | Future    |
| Embedded Repository    | Future    |

Sprint 002 implements the **Standard Repository** only.

---

# 5. Repository Root

Every generated repository shall contain:

```text
my-workshop/

├── repository/
├── workspace/
├── packages/
├── cache/
├── logs/
├── exports/
├── settings/
│
├── README.md
├── .gitignore
├── repository.json
└── workspace.json
```

No additional top-level directories shall be generated unless defined by an approved specification.

---

# 6. Root Directory Responsibilities

## repository/

Contains Engineering Objects and engineering knowledge.

This directory is the authoritative engineering datastore.

---

## workspace/

Contains user-specific workspace information.

Examples:

* Window layouts
* Open tabs
* Session state
* Recent files

Workspace data is not engineering knowledge.

---

## packages/

Contains locally installed OEP packages.

Packages extend repository capability without modifying Foundation.

---

## cache/

Contains disposable generated data.

May be safely deleted.

---

## logs/

Contains diagnostic information and application logs.

No engineering knowledge shall be stored here.

---

## exports/

Contains generated artifacts.

Examples:

* PDF
* Images
* Reports
* Certificates

Generated files may be recreated.

---

## settings/

Contains repository configuration.

Examples:

* User preferences
* Studio configuration
* Repository settings

---

# 7. Root Files

## README.md

Repository overview.

---

## repository.json

Contains repository metadata including:

* Repository ID
* OEP Version
* Template Version
* Creation Date

---

## workspace.json

Contains workspace configuration including:

* Default Studio
* Theme
* Workspace Version

---

## .gitignore

Generated according to Foundation standards.

---

# 8. Generator Requirements

The CLI shall generate:

* Repository directory structure
* Placeholder README files
* Metadata files
* Configuration files

The CLI shall not generate engineering content.

---

# 9. Naming Rules

Directory names shall be:

* Lowercase
* Stable
* Predictable
* Cross-platform compatible

Generated identifiers shall be globally unique.

---

# 10. Future Extension Points

Future specifications may define:

* Enterprise repositories
* Multi-user repositories
* Cloud synchronization
* Distributed repositories
* Registry synchronization

These extensions shall not invalidate the base repository structure.

---

# 11. Acceptance Criteria

Sprint 002 is complete when:

* `oep init <repository-name>` creates the repository exactly as specified.
* All required directories are generated.
* All required files are generated.
* Metadata files are valid.
* No undocumented files are created.
* Repository generation is deterministic across supported platforms.

---

# 12. Engineering Principle

Every OEP repository should be immediately recognizable regardless of who created it or on which operating system it was generated.

---

# 13. Implementation Boundary

This specification defines **what** is generated.

It does not define **how** the CLI performs generation.

Implementation details remain the responsibility of the CLI.

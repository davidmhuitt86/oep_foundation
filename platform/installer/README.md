# Installer

Status: Foundation (WP-REP-001 install slice + WP-REP-002 Repository Registry & Lifecycle + WP-REP-004 Trust & Signing)

Purpose: Installs a validated `.oep` package archive (PKG-001/PKG-002) into an open Foundation Repository — extracting its Repository Fragment's Engineering Objects and Relationships, recording the install in the Repository Registry (the authoritative inventory of installed engineering packages, per PKG-008's framing), and leaving `platform/runtime`'s `FoundationRuntime` to actually mutate the repository (this module never touches `ObjectStore`/`RelationshipStore` itself; see below).

See `OEP-ARCH-002-REPOSITORY_RUNTIME_ASSESSMENT.md` (`specifications/architecture/`) for the full architectural review and roadmap this Work Package implements the first slice of.

## Architecture

- `include/oep/installer/zip_reader.hpp` — public `ZipReader::open`/`has_entry`/`entry_names`/`read_bytes`/`read_text`. Reads only the ZIP "Stored" (uncompressed) entry method — see its own doc comment for why DEFLATE support is a deliberately deferred scope limit, not an oversight.
- `include/oep/installer/oep_package_manifest.hpp` — public `OepPackageManifest` and `parse_oep_package_manifest`, PKG-002's manifest shape and validation rules (distinct from `oep::packages::PackageManifest`, OEP-SPEC-010's unrelated local-package manifest — see OEP-ARCH-002 §0).
- `include/oep/installer/package_installer.hpp` — public `extract_package`: opens an archive, validates its manifest, and parses its Repository Fragment (`fragment/objects/*.json`, `fragment/relationships/*.json` — renamed from `repository/` in WP-REP-002, per OEP-ARCH-002 §0.2(b); an archive still using the legacy `repository/` directory name is also accepted) into Foundation's own `EngineeringObject`/`Relationship` types, in the same camelCase JSON convention `ObjectStore`/`RelationshipStore` already read and write. Pure parsing only — never opens or mutates a repository.
- `include/oep/installer/repository_registry.hpp` — public `RepositoryRegistry` (`record_install`/`is_installed`/`list_installed`/`find_owner`/`search_packages`) and `RepositoryRegistryEntry` (renamed and expanded from WP-REP-001's `PackageRegistry`/`InstalledPackageRecord` in WP-REP-002): one JSON record per installed package under `<repository>/packages/<packageId>/registry.json`, now carrying full manifest metadata (title/summary/category/schema version/engineering domains), publisher, installation date and path, install source, a SHA-256 package hash, a runtime state (always `"Installed"` through WP-REP-002 — other PKG-008 §6 states require update/uninstall/activation, all out of scope), and the contributed object/relationship IDs.
- `include/oep/installer/sha256.hpp` — hand-rolled SHA-256 (FIPS 180-4), tested against NIST vectors, consistent with this codebase's no-dependency convention. An integrity fingerprint for the Repository Registry's Package Hash field only — **not** a trust or signature mechanism.
- `include/oep/installer/sha512.hpp` — hand-rolled SHA-512 (FIPS 180-4), tested against NIST vectors. Exists solely because Ed25519 (RFC 8032) is defined over SHA-512; not used for package hashing (that stays SHA-256).
- `include/oep/installer/ed25519.hpp` — hand-rolled Ed25519 signature **verification** (RFC 8032), tested against the RFC's own published test vectors. Deliberately verify-only: Foundation never signs anything and never holds a private key — signing is `@oep-exchange/signing`'s job (Node's built-in `node:crypto`), an independent implementation on the other side of the trust boundary. Constant-time execution is not attempted, since verification handles only public data (message, public key, signature) — that requirement applies to signing/secret-key code, which does not exist here.
- `include/oep/installer/trust_store.hpp` — public `TrustStore` (WP-REP-004, PKG-005): locally trusted publisher certificates (`add_certificate`/`get_certificate`/`list_certificates`/`revoke_certificate`) and a per-repository signature policy (`get_policy`/`set_policy`), persisted under `<repository>/settings/trust/`. Entirely offline — no certificate authority, no online revocation feed (PKG-005 §3).
- `include/oep/installer/package_verifier.hpp` — public `verify_package_trust` (WP-REP-004): the PKG-005 §11 verification pipeline (structure → content hashes → certificate → signature → state) for one archive against a `TrustStore`, producing a `TrustState` (`Trusted`/`Unsigned`/`UnknownPublisher`/`ExpiredCertificate`/`RevokedCertificate`/`InvalidSignature`/`Tampered`).
- `include/oep/installer/repository_registry.hpp` — public `RepositoryRegistry` (`record_install`/`is_installed`/`list_installed`/`find_owner`/`search_packages`) and `RepositoryRegistryEntry` (renamed and expanded from WP-REP-001's `PackageRegistry`/`InstalledPackageRecord` in WP-REP-002): one JSON record per installed package under `<repository>/packages/<packageId>/registry.json`, now carrying full manifest metadata (title/summary/category/schema version/engineering domains), publisher, installation date and path, install source, a SHA-256 package hash, a runtime state (always `"Installed"` through WP-REP-002 — other PKG-008 §6 states require update/uninstall/activation, all out of scope), the trust outcome recorded at install time (`trust_status`/`trust_fingerprint`, WP-REP-004), and the contributed object/relationship IDs.
- `src/zip_reader.cpp`, `src/oep_package_manifest.cpp`, `src/sha256.cpp`, `src/sha512.cpp`, `src/ed25519.cpp`, `src/trust_store.cpp`, `src/package_verifier.cpp`, `src/package_installer.cpp`, `src/repository_registry.cpp` — implementation.

Consumed exclusively by `platform/runtime`'s `FoundationRuntime` (`install_package`/`list_installed_packages`, WP-REP-002's lifecycle queries `get_installed_package`/`find_package_owner`/`verify_package`/`search_installed_packages`, and WP-REP-004's `trust_add_certificate`/`trust_get_certificate`/`trust_list_certificates`/`trust_revoke_certificate`/`trust_get_policy`/`trust_set_policy`), which is in turn the only way `platform/cli`'s `oep package`/`oep trust` subcommands and the Public C API's `oep_package_*`/`oep_trust_*` functions touch package installation and trust — this module, like every other Foundation subsystem, is reached exclusively through the Runtime (OEP-SPEC-011 §6).

## Trust & Signing (WP-REP-004)

`FoundationRuntime::install_package` now verifies trust **before opening any Repository Transaction** (this Work Package's explicit requirement): `verify_package_trust` opens the archive, requires `signatures/package.sig` (a JSON block: `algorithm`, `publisherId`, `publicKeyHex`, `signatureHex`, and a `contentHashes` list covering every archive entry outside `signatures/`, per PKG-005 §9/§10 — nothing in the payload is exempt), checks every listed hash against the archive's actual bytes (a mismatch or an uncovered entry is `Tampered`), looks up the publisher in the repository's `TrustStore` (`UnknownPublisher` if absent, `RevokedCertificate`/`ExpiredCertificate` if applicable), and finally verifies the Ed25519 signature over the canonical payload (sorted `<path>\n<sha256>\n` pairs — the one piece of format the C++ verifier and the TypeScript signer must agree on byte-for-byte, documented identically in both places since they're independent implementations in different languages).

An archive that is `Tampered`, `InvalidSignature`, `UnknownPublisher`, `ExpiredCertificate`, or `RevokedCertificate` is rejected outright — nothing is extracted, no transaction is ever opened. An `Unsigned` archive installs exactly as in WP-REP-001–003 **unless** the Trust Store's policy requires signatures (`TrustStore::set_policy(true)`), which defaults to `false` — preserving every earlier work package's default unsigned-install behavior.

**A real bug was caught and fixed while first validating this against a genuinely signed archive** (not a hand-crafted test fixture): `@oep-exchange/signing`'s file-sort used `String.prototype.localeCompare`, which orders paths differently than C++'s `std::string::operator<` for certain names (e.g. `"LICENSE.md"` vs. `"licenses/LICENSE.md"`) — a real signed `engineering-demo.oep` installed as `InvalidSignature` until the signer's sort was changed to plain byte-wise comparison. See `oep_exchange/packages/signing`'s own notes and its regression test.

## What WP-REP-001 deliberately does not implement

Named explicitly rather than silently absent, per this Work Package's own scope:

- **Dependency resolution** (PKG-004) — not evaluated. A package declaring dependencies installs as if it had none.
- **Cross-package merge/ownership logic** (PKG-006) — `extract_package`/`FoundationRuntime::install_package` handle one package's own Repository Fragment in isolation; there is no conflict detection or identity reconciliation against content another package already installed.
- ~~**Signature verification**~~ — **implemented in WP-REP-004** (Trust & Signing): see the section above.
- ~~**Transactions**~~ — **superseded by WP-REP-003 (Repository Transaction Engine)**: `FoundationRuntime::install_package` is now atomic. A failure partway through rolls back every object/relationship the install had created, and the whole install is journaled under `<repository>/logs/transactions/` — see `platform/runtime/README.md`. This module itself is still transaction-free by design (it only parses; the Runtime owns atomicity).
- **Networking** — installation is local-filesystem-to-local-repository only; there is no HTTP surface here (see OEP-ARCH-002 §4.2's still-open question).
- **Update and uninstall** — `RepositoryRegistry::record_install` fails outright if a package with the same `packageId` is already recorded, regardless of version. There is no update or removal path, and `runtime_state` never leaves `"Installed"`.
- **DEFLATE-compressed archives** — see `ZipReader`'s own doc comment. A `.oep` archive built with compression enabled (the default for most ZIP-writing tooling, including `oep_exchange`'s `oep-package build` today) cannot currently be installed; only Stored/uncompressed archives can. `engineering-demo.oep` was rebuilt with Stored compression to be installable by this Work Package — see its own regeneration notes in `oep_exchange`.

All of the above are named, scoped follow-on Work Packages in OEP-ARCH-002 §5 (WP-REP-002 through WP-REP-005), not gaps discovered after the fact.

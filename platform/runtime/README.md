# Runtime

Status: Foundation (Foundation Runtime)

Purpose: The single entry point through which applications interact with Foundation, per OEP-SPEC-011-FOUNDATION_RUNTIME. Coordinates lifecycle and service access for Repository, Search, Graph, Validation, Package Manager, and Archive without reimplementing any of them.

## Architecture

- `include/oep/runtime/foundation_runtime.hpp` — public `FoundationRuntime` (`initialize` / `open_repository` / `close_repository` / `shutdown`, Repository Context accessors, Service Registry accessors, `export_repository` / `import_repository`, `create_template` / `list_templates` / `instantiate_template`, `execute_batch_create` / `execute_batch_delete` / `validate_batch_create`, — per Work Package 014 — `create_object` / `update_object` / `delete_object`, `create_relationship` / `update_relationship` / `delete_relationship`, `begin_transaction` / `commit_transaction` / `rollback_transaction` / `transaction_active`, `batch_create_objects` / `batch_create_relationships`, and — per WP-REP-001 (Repository Runtime) — `install_package` / `list_installed_packages` / `package_registry`) and `RuntimeState`
- `src/foundation_runtime.cpp` — deterministic state transitions and subsystem construction/registration

`open_repository` establishes where an opened repository's Engineering Objects, Relationships, and Audit Events live on disk: `repository/objects/`, `repository/relationships/`, and `repository/audit/` beneath the repository root, with packages discovered under the repository's existing top-level `packages/` directory. This convention is not yet wired into `oep init`/the Foundation Generator.

Only one repository may be open at a time. Every service accessor (`object_store`, `relationship_store`, `audit_store`, `search_engine`, `graph_engine`, `validator`, `package_manager`) returns `nullptr` unless a repository is currently open — before initialization, before opening, after closing, and after shutdown all behave identically in this respect. `export_repository` and `create_template` require an open repository (they capture *the* open one); `import_repository`, `list_templates`, and `instantiate_template` do not — they target a destination/template-store directory to create or read, not one to open — but all five remain exposed on `FoundationRuntime` and delegate entirely to `oep::archive` (renamed from `oep::exchange` in WP-REP-002 — see `platform/archive/README.md`), so every repository-touching operation executes through the Runtime. `execute_batch_create`, `execute_batch_delete`, and `validate_batch_create` all require an open repository and delegate entirely to `oep::repository::BatchProcessor`.

`platform/api`'s Public C API (Work Package 012: Engineering Object Enumeration, Repository Statistics; Work Package 013: Engineering Relationship Enumeration, Repository Search) added no new surface to `FoundationRuntime` itself — every function those work packages exposed was built entirely from the existing `object_store()`/`relationship_store()`/`search_engine()`/`current_metadata()`/`current_package_set()` accessors already described above, translated into fixed-layout C structures at the API boundary. Work Package 013's search functions, in particular, read the same `SearchEngine` instance `search_engine()` already returns — the one built by `build_index` during `open_repository` — rather than constructing or indexing anything new.

**Work Package 014** is the first to extend `FoundationRuntime`'s own interface for the Public C API's benefit: `create_object`/`update_object`/`delete_object` and `create_relationship`/`update_relationship`/`delete_relationship` were added because `object_store()`/`relationship_store()` are read-only-oriented accessors (they return `const ObjectStore*`/`const RelationshipStore*`) and Work Package 014's architectural guidance requires "every Public C API mutation function shall delegate through Foundation Runtime" — the Public API must never reach a store directly, so Runtime needed its own mutation entry points to delegate through. Each new method is a thin orchestration layer over the corresponding `ObjectStore`/`RelationshipStore` method; no validation logic is duplicated.

Also new in Work Package 014: `begin_transaction`/`commit_transaction`/`rollback_transaction`/`transaction_active`, and `batch_create_objects`/`batch_create_relationships` built on top of them. A transaction is implemented as an in-memory, per-`FoundationRuntime`-instance undo log (`TransactionLogEntry`, private) — `ObjectStore`/`RelationshipStore` gained no new persistence concept; every mutation still writes to disk immediately, transaction or not. `rollback_transaction` replays the log in reverse using the stores' own `remove`/`update`/`restore` methods (the same `restore` path Import and Templates already use to reconstruct exact prior state). `reset_repository_context()` (called by both `close_repository()` and `shutdown()`) now rolls back an active transaction first, so a repository is never left with a transaction stranded open when it closes.

## Package Installation (WP-REP-001, Repository Runtime)

`install_package(const std::filesystem::path& archive_path)` is the first vertical slice of the Foundation Repository Runtime: it opens a `.oep` archive via `oep::installer::extract_package` (built on `oep::installer::ZipReader`, which supports only the ZIP "Stored"/uncompressed entry method — DEFLATE is rejected), then, for every Repository Fragment object and relationship the archive contains, calls the existing `ObjectStore::create`/`RelationshipStore::create` unchanged — archive-declared IDs are preserved rather than remapped, since both stores already keep a caller-supplied ID when one is present. On completion it rebuilds the Search and Graph indexes (`search_->build_index`/`graph_->build_graph`, the same calls `open_repository` itself makes) and records one entry via the new `oep::installer::PackageRegistry` (`package_registry()` accessor; persisted at `<repository>/packages/<packageId>/registry.json`, which the pre-existing `oep::packages::PackageManager` never matches — it only discovers files literally named `package.json`, so the two "package" concepts coexist without collision, per OEP-ARCH-002 §4.1).

**`install_package` is atomic since WP-REP-003 (Repository Transaction Engine)** — superseding WP-REP-001/002's deliberately-non-transactional behavior: the whole install runs inside one Repository Transaction; any failure rolls back every create performed so far (covered by `test_a_failure_partway_through_is_rolled_back` in `tests/runtime/package_installation_tests.cpp`, the inversion of the earlier no-rollback test) and the transaction is journaled either way. See "Repository Transaction Engine" below.

`list_installed_packages()` returns every Repository Registry record for the currently open repository, delegating entirely to `RepositoryRegistry::list_installed()`.

Both methods require an open repository, following the same precondition every other repository-touching `FoundationRuntime` method already enforces; `registry_` (the `RepositoryRegistry` instance) is constructed in `open_repository()` alongside the other per-repository services and reset in `reset_repository_context()`, matching the lifecycle of `object_store_`/`relationship_store_`/etc. exactly.

## Package Lifecycle Queries (WP-REP-002)

WP-REP-002 renamed WP-REP-001's minimal Package Registry to the **Repository Registry** (`oep::installer::RepositoryRegistry`, accessor `repository_registry()`), expanded its per-package record to the full WP-REP-002 §4 field set (manifest metadata, publisher, installation date/source/path, SHA-256 package hash, runtime state, contribution IDs), and added four read-only lifecycle query services — none of which mutate anything, since update/uninstall/activation remain explicitly out of scope:

- `get_installed_package(package_id)` — the full Repository Registry entry; "not installed" is a normal result, not an error.
- `find_package_owner(entity_id)` — which installed package contributed a given Engineering Object or Relationship.
- `verify_package(package_id)` — confirms every recorded contribution still exists in the open repository's own stores, and (when the source archive still exists at its recorded path) that its bytes still hash to the recorded SHA-256. A deleted archive is a reported condition, not a failure — the archive is transport, not the installed content.
- `search_installed_packages(query)` — case-insensitive substring match over registry metadata plus the *live* names of each package's installed Engineering Objects (cross-referenced against `ObjectStore` at query time — object names are never duplicated into the registry, per CLAUDE.md's "Never duplicate Engineering Object data").

`install_package` now also records the archive's absolute path and SHA-256 hash and sets `runtime_state = "Installed"` at install time. The hash is an integrity fingerprint only, not a trust mechanism — signature verification is PKG-005 / a future work package.

## Repository Transaction Engine (WP-REP-003)

WP-REP-003 upgraded Work Package 014's transaction primitives (`begin_transaction`/`commit_transaction`/`rollback_transaction`/`transaction_active` — every signature unchanged) into the Repository Transaction Engine, implementing the install-scope subset of PKG-003:

- **Identity and journaling** — every transaction has a UUIDv4 id and, on close, writes one permanent record (`TransactionRecord`: state, description, open/close timestamps, and one `JournalEntry` per operation with its target and previous/new state) to the Transaction Journal (`transaction_journal.hpp/.cpp`, persisted under `<repository>/logs/transactions/` — inside the `logs/` directory OEP-SPEC-002 §5 already reserves). The journal records what happened; Work Package 014's in-memory compensating-action undo log — reused unchanged — is what performs a rollback.
- **Every write executes through a transaction** — a mutation called outside an explicit transaction runs inside an *implicit* one, begun and committed automatically around the single operation, so it is journaled like any other write. `install_package` is atomic: one transaction per install, rolled back completely on any failure (including a failed Repository Registry write), journaled Committed or RolledBack. An install inside a caller's explicit transaction is rejected (no nesting, per Work Package 014).
- **States** — `Opened`/`Committed`/`RolledBack`/`Failed` (a rollback whose undo could not fully complete), a deliberate subset of PKG-003 §5; the full lifecycle (Planning/WaitingForApproval/Indexing/Activating/...) belongs to capabilities this Work Package excludes (dependency resolution, previews, activation).
- **One documented behavior change** — `commit_transaction` can now fail in exactly one new way: the journal record could not be written. The mutations are already persisted in that case; the error exists so the caller knows the audit trail is incomplete, never to signal data loss. A journal problem never prevents a rollback from restoring repository state — restoring data always outranks recording history.
- **Queries** — `current_transaction_info()` (active transaction's id/description/entry count), `transaction_history()` (every journaled record, sorted), `get_transaction_record(id)`.
- **Documented boundaries** — `execute_batch_create`/`execute_batch_delete` (OEP-SPEC-020's `BatchProcessor`) retain their own validate-everything-then-execute atomicity and do not flow through this engine; `import_repository`/`instantiate_template` construct *new* repositories rather than writing to the open one, so there is no open-repository state for a transaction to protect. Crash-recovery/repair from the journal is PKG-003 §21 territory, deferred alongside update/uninstall.

The CLI's `oep object create|delete` and `oep relationship create|delete` were migrated in this Work Package from direct `ObjectStore`/`RelationshipStore` access to the Runtime's own mutation methods, so CLI writes are journaled exactly like Public-C-API writes.

## Trust & Signing (WP-REP-004)

`FoundationRuntime` gained a `TrustStore` accessor (`trust_store()`) and six pass-through methods — `trust_add_certificate`/`trust_get_certificate`/`trust_list_certificates`/`trust_revoke_certificate`/`trust_get_policy`/`trust_set_policy` — over `oep::installer::TrustStore` (WP-REP-004, PKG-005). These are local, offline, user-directed writes to `<repository>/settings/trust/`, not Repository writes, so they never open a Repository Transaction.

**`install_package` now verifies trust BEFORE any Repository Transaction begins** (this Work Package's explicit, load-bearing requirement): immediately after confirming the package isn't already installed, and strictly before `open_transaction_internal` is ever called, `oep::installer::verify_package_trust` runs against the repository's `TrustStore`. A `Tampered`, `InvalidSignature`, `UnknownPublisher`, `ExpiredCertificate`, or `RevokedCertificate` result rejects the install immediately — no archive content is extracted, no transaction is opened, nothing is journaled. Proven by a dedicated test (`test_signed_but_untrusted_publisher_is_rejected` in `tests/runtime/trust_integration_tests.cpp`) that asserts `transaction_history()` is empty after a trust-rejected install. An `Unsigned` archive proceeds exactly as in WP-REP-001–003, unless the Trust Store's policy requires signatures.

On a successful install, the resolved `TrustState` and (when a certificate matched) its fingerprint are recorded on the `RepositoryRegistryEntry` (`trust_status`/`trust_fingerprint`) and returned via `RuntimeInstallResult::trust_status` — queryable afterward without re-verifying (PKG-005 §16's "revalidate at any time" is deferred; `oep package verify`'s existing WP-REP-002 hash/contribution checks are unrelated to trust and untouched).
